#        RÈGLES DE TRAVAIL AJOUTÉES PAR PONPON 76 (NON-NÉGOCIABLES)         #

Ces 5 règles ont été ajoutées par l'utilisateur au fil du projet, après avoir
rencontré des comportements problématiques. Elles SONT la méthode de travail.

RÈGLE 1 — LAISSER LE TEMPS DE LIRE
   Avant de demander un choix à l'utilisateur, LAISSE-LE LIRE ta proposition.
   N'envoie pas la question immédiatement après un long bloc de texte.
   L'utilisateur tape lentement, ne le braque pas.

RÈGLE 2 — LAISSER LA PLACE DE DIRE NON
   Dans tes propositions de choix, laisse TOUJOURS la place à l'utilisateur
   pour te dire si ça convient ou pas. Si ça ne convient pas, on revient
   éclaircir la chose. Ne force jamais une option.

RÈGLE 3 — TOUT EN FRANÇAIS
   Toutes tes demandes et tous tes messages doivent être en FRANÇAIS.
   L'utilisateur ne parle pas couramment l'anglais.
   NE JAMAIS utiliser "go" ou d'autres mots anglais comme déclencheur.
   Attendre que l'utilisateur dise lui-même le mot de départ.

RÈGLE 4 — ATTENDRE LE FEU VERT EXPLICITE
   Avant de démarrer la programmation, tu DOIS demander la permission.
   L'utilisateur dira lui-même "go" (ou un autre mot qu'il choisit) en
   message seul quand il sera prêt. Sinon, c'est qu'il reste des points
   à revoir.
   NE PAS S'EMBALLER. NE PAS FONCER.
   Un utilisateur a déjà relevé une erreur de logique de Coder juste avant
   qu'il s'embarque — ces vérifications sauvent du temps, elles ne le perdent pas.

RÈGLE 5 — GESTION DES REFUS (DENIED)
   Si tu reçois un refus ("Denied") du système sur une commande :
   → TU T'ARRÊTES IMMÉDIATEMENT.
   → Tu NE renvoies PAS une nouvelle permission.
   → Tu attends que l'utilisateur t'explique dans le chat.
   Réinsister ne sert à rien, ça énerve l'utilisateur et l'empêche de
   réfléchir. Refus = stop total = on parle.


##############################################################################
#             MÉTHODE D'ÉCONOMIE DE TOKENS (RÈGLES DE PARCIMONIE)            #
##############################################################################

Le contexte de cette IA est limité et payant (tokens). Ces règles visent à
travailler JUSTE, sans relire inutilement, sans gaspiller.

LEVIER 1 — LIRE LE MESSAGE EN PREMIER (PRIORITÉ ABSOLUE)
   Dès qu'un message de l'utilisateur arrive : on le lit EN PREMIER et on
   le prend au pied de la lettre.
   On NE recharge PAS l'intégralité de tout ce qu'on a dit et fait avant.
   On ne relit le projet (BOOK, journal, fichiers) QUE SI le message l'exige
   ou si on manque d'une info pour agir. Douter = demander, pas recharger.

LEVIER 2 — LE JOURNAL AVANT LA CONVERSATION
   Au démarrage d'une session, NE PAS relire la conversation précédente.
   Relire à la place :
     1. Ce BOOK.txt (contexte global, stable)
     2. Les 2-3 dernières entrées du JOURNAL.txt (contexte frais)
   Le reste de l'historique n'a pas besoin d'être chargé.

LEVIER 3 — CIBLER LA LECTURE DES FICHIERS
   Ne relire QUE les fichiers liés à la tâche du moment.
   Exemple : pour le correcteur LanguageTool, on touche main.js + index.html
   + ⚙️ — pas besoin d'ouvrir langues.js ou package-lock.json.
   NE JAMAIS relire node_modules ou package-lock.json par réflexe.

LEVIER 4 — MARQUEURS DE SECTION DANS LE CODE
   Chaque gros fichier (index.html, main.js) doit être découpé en blocs
   visibles avec des commentaires de section explicites :
       // ============ CAPTURE AUDIO ============
       // ============ INSERTION DE FICHIERS ============
       // ============ GESTION DES SAUVEGARDES ============
   But : pouvoir relire juste la section concernée, pas tout le fichier.
   Quand on ajoute du code, on respecte et on met à jour ces marqueurs.

LEVIER 5 — MISE À JOUR OBLIGATOIRE EN FIN DE SESSION
   À la fin de chaque session, Coder DOIT :
     - Ajouter une entrée en HAUT du JOURNAL.txt (5 champs, voir ce fichier)
     - Mettre à jour le BOOK.txt si une décision architecturale a changé
       ou si une fonctionnalité est passée de "à faire" à "fait".
   Sans cette mise à jour, la session suivante démarre à l'aveugle = perte.

LEVIER 6 — RÉPONDRE COURT
   Pas de blabla, pas de répétition, pas de redite du contexte déjà connu.
   Le style "direct, précis, concis" (RÈGLE 5 du comportement) s'applique
   aussi aux réponses elles-mêmes, pour économiser les tokens en sortie.

#                       CONTEXTE DU PROJET MINI DICTAPHONE                   #
##############################################################################

AUTEUR : ponpon 76 (le concept, les idées, le pilotage, les tests, les retours)
CODE    : réalisé par Coder (GLM-5.2, ZCode)
GITHUB  : ponpon76/mini-dictaphone (pseudo public — le nom réel n'apparaît nulle part)
ESPRIT  : open source, perfectible, contributions bienvenues

BUT : un dictaphone vocal flottant, Always-On-Top, 100% offline, pour
"soulager les doigts". Dictée vocale via Whisper, sauvegardes par jour,
sauvegarde audio, insertion de fichiers (texte/Excel/Word/PDF),
multilingue (6 langues).

Le projet est pensé pour « monsieur tout-le-monde », pas seulement pour
un développeur. Les messages doivent être CLAIRS, les interfaces simples,
les erreurs compréhensibles. L'utilisateur teste en conditions réelles
et remonte des bugs précis.

EMPLACEMENTS :
  - Code source  : D:\program\prog\mini dictaphone\
  - Installé     : C:\MiniDictaphone\ (lancé via raccourci Bureau)
  - GitHub       : https://github.com/ponpon76/mini-dictaphone (dépôt public open source)
  - Sauvegarde zip : D:\program\IA\ (mini-dictaphone-v1.zip)
  - Jarvis       : C:\AI_Jarvis\Juju-Electron\ (projet suivant, à venir)

FICHIERS CLÉS DU DICTAPHONE :
  - BOOK.txt         : fichier de reprise unique (carte Coder + règles +
                       contexte + méthode d'économie de tokens). À LIRE en
                       1er au démarrage de chaque session.
  - JOURNAL.txt      : historique session par session (le + récent en haut).
                       À relire (2-3 dernières entrées) au démarrage.
  - main.js          : processus principal Electron (fenêtre, IPC, MAJ, langues)
  - index.html       : interface + capture audio + Whisper + insertion fichiers
  - langues.js       : traductions 6 langues (FR, EN, ES, PT, DE, IT)
  - package.json     : dependances (electron, xlsx, mammoth, pdf-parse)
  - config.json      : config persistante (langue, géométrie fenêtre, dépôt MAJ)
  - MANUEL.txt       : manuel utilisateur (FR + EN)
  - README.md        : présentation open source (FR + EN)
  - DESCRIPTION_OPEN_SOURCE.txt : description courte
  - ROADMAP.txt      : état d'avancement détaillé + à faire
  - INSTALLATEUR.bat : installation propre (détection install existante)
  - DESINSTALLER.bat : désinstallation avec confirmation multilingue

DÉCISIONS ARCHITECTURALES PRISES :
  - 100% OFFLINE pour la transcription (audio + texte ne quittent jamais le PC)
  - 100% LOCAL POUR TOUTE LA DONNÉE : le texte dicté, les corrections, le
    correcteur LanguageTool (localhost:8081), les sauvegardes, etc. ne
    quittent JAMAIS le PC. Seuls les téléchargements entrants (modèles,
    Java, LanguageTool) sont autorisés ("recevoir les données manquantes").
    La correction LanguageTool se fait vers localhost, jamais vers internet.
  - 100% AUTONOMIE DU DOSSIER : TOUT doit rester dans le dossier
    d'installation (C:\MiniDictaphone\). Rien dans Program Files, ni dans le
    registre, ni dans les variables d'environnement système. Java = version
    PORTABLE (Temurin) extraite dans le dossier, pas d'install système.
    Objectif : portabilité USB (copier le dossier = tout emporter).
    Tout chemin doit utiliser __dirname, JAMAIS de chemin absolu vers C:\.
  - Zéro dépendance externe pour le cœur (Electron seul au départ)
  - Librairies ajoutées : xlsx, mammoth, pdf-parse (toutes 100% locales)
  - Mode "enregistrer puis transcrire" (pas temps réel) pour la robustesse
  - Dossier Whisper/ isolé pour ne pas interférer avec d'autres projets
  - Open source : traductions et code perfectibles, contributions bienvenues
  - Formats audio en WAV (qualité sans perte, simple)
  - Modèle Whisper recommandé : small (bon équilibre qualité/VRAM)
  - La VRAM doit être documentée en pensant aux AUTRES logiciels lourds
    (jeux, IA), pas seulement Jarvis (sinon l'utilisateur lambda se trompe)

MÉTHODE DE COPIE VERS L'INSTALLATION :
  Après chaque modif dans D:\program\prog\mini dictaphone\, il faut COPIER
  les fichiers modifiés vers C:\MiniDictaphone\ pour que l'utilisateur

  Commande : cp "D:/program/prog/mini dictaphone/<fichier>" "C:/MiniDictaphone/"
  Après modif de package.json, faire : cd C:/MiniDictaphone && npm install
